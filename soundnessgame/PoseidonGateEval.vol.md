~~## 1. `_mdsPartialLayerInit` Potential Matrix Transposition (Soundness-Critical â€” Requires Verification)~~
> Fixed in round 1

**Description:**


This computes `result[c] = ÎŁ_r state[r] * M[c-1][r-1]`, i.e., standard matrix-vector multiply where `M[output][input]` (output-row-first layout).

**Affected lines:** 93â€“95 in `_mdsPartialLayerInit`.

**Why this is a soundness concern:**
The Plonky2 Rust reference loop is structured as:
```rust

    for c in 1..12 {     // c = output index
        result[c] += state[r] * FAST_PARTIAL_FIRST_ROUND_CONSTANT[r-1][c-1];

```
That stores and accesses the matrix as `M[input][output]`, computing `result[c] = ÎŁ_r state[r] * M[r-1][c-1]`.

`PoseidonConstants.initialMatrix` uses the signature `(uint256 row, uint256 col)` with `idx = row * 11 + col`. If it was generated by directly mapping Rust's `FAST_PARTIAL_FIRST_ROUND_CONSTANT[r][c]` â†’ `initialMatrix(r, c)` (preserving input-first order), the Solidity call `initialMatrix(c-1, r-1)` reverses the indices â€” using column `r-1` as the row â€” implementing `M^T` instead of `M`.

The matrix is **not symmetric** (independently verified from the constants: `M[0][1]` = `0xdc927721da922cf8` â‰  `M[1][0]` = `0xe796d293a47a64cb`), so a transposition implements a cryptographically different permutation.

**Consequence:** If transposed, the verifier accepts proofs for a *different* Poseidon hash function than the prover uses. An adversary who discovers this can forge Merkle tree proofs (used for STARK soundness), completely breaking proof validity. Valid Plonky2 proofs would also be rejected, making the system non-functional.

**Suggested fix:** Cross-check by running a known-answer test with Plonky2's reference Poseidon implementation (input: 12 zeros â†’ compare output state). If broken, change line 94 to `PoseidonConstants.initialMatrix(r - 1, c - 1)`.

~~## 2. `_wireFullSbox1` Completely Lacks Parameter Bounds Checking~~
> Fixed in round 1

**Description:**
```solidity

    return 87 + 12 * round + i;
}
```
No check on `round` or `i`, unlike `_wireFullSbox0` which validates `round >= 1 && round < HALF_N_FULL`. With `round >= 4` or `i >= 12`, the returned index exceeds 134.

**Affected lines:** 38â€“40 (`_wireFullSbox1`), compared to 33â€“36 (`_wireFullSbox0`).

**Why this is a soundness concern:**
The only protection is `require(wires.length >= 135)` at line 123, which is a *lower* bound only (see Issue 3). If a caller passes `wires.length > 135`, an out-of-bounds `round` or `i` silently reads from extra slots outside the intended wire layout â€” the constraint evaluation accesses the wrong wire, and the wrong value enters the S-box chain. All downstream constraints are then evaluated against corrupted state, potentially rendering all 48 second-half sbox constraints trivially satisfiable with fabricated witnesses.

**Suggested fix:** Add `require(round < HALF_N_FULL && i < WIDTH, "wireFullSbox1: out of range");` to match `_wireFullSbox0`'s approach.

~~## 3. `wires` Array Length Checked Only From Below (No Upper Bound)~~
> Fixed in round 1

**Description:**
```solidity
// PoseidonGateEval.sol:123
require(wires.length >= 135, "PoseidonGate: wires too short");
```

The wire layout is fixed at exactly 135 wires. Passing `wires.length > 135` is silently accepted.

**Affected line:** 123.

**Why this is a soundness concern:**
Combined with Issue 2, a caller supplying `wires` with 150+ entries passes validation, enabling `_wireFullSbox1` to access indices 135â€“149 with unvalidated `round`/`i` values. Those extra slots are under the caller's control and can contain arbitrary field elements. An adversary could craft a `wires` array where the first 135 entries satisfy trivial constraints while the actual second-half S-box constraints are evaluated against attacker-chosen values in the overflow region.

**Suggested fix:** Change the check to `require(wires.length == 135, "PoseidonGate: wrong wire count");`.

~~## 4. `_wireFullSbox0` Validates `round` But Not Column Index `i`~~
> Fixed in round 1

**Description:**
```solidity

    require(round >= 1 && round < HALF_N_FULL, "wireFullSbox0: round out of range");

```

**Affected lines:** 33â€“36.

**Why this is a soundness concern:**

The prover could then satisfy what appears to be a round-1 full-sbox constraint by manipulating a partial-sbox wire, breaking the independence of the two constraint families.

**Suggested fix:** Add `require(i < WIDTH, "wireFullSbox0: i out of range");`.
